# 文档管理规范 (Documentation Standards)

为了维护项目文档的清晰性、准确性和可维护性，所有团队成员（包括 AI 助手）应遵循以下规范。

## 1. 文档分类与目录结构

所有文档应统一存放在 `docs/` 目录下，严禁在项目根目录散落文档（README.md 除外）。

*   **`docs/architecture/`**: 架构设计、技术方案、RFC。
*   **`docs/api/`**: API 接口定义、第三方服务集成文档。
*   **`docs/guides/`**: 操作手册、部署指南、开发环境配置。
*   **`docs/test_reports/`**: 测试计划、测试报告。
*   **`docs/archive/`**: 归档文件。
    *   `issues/`: 针对特定 bug 的分析与修复记录（任务完成后移入）。
    *   `plans/`: 已过期的实施计划。
    *   `reports/`: 历史状态报告。

## 2. 命名规范

*   **文件名**: 建议使用英文，单词间用连字符 `-` 或下划线 `_` 分隔（如 `api-integration-guide.md`）。若使用中文，应简洁明了（如 `部署文档.md`）。
*   **标题**: 文档必须包含一级标题 (`# Title`)。

## 3. 文档生命周期

1.  **创建**: 新建文档时，应根据内容放入对应目录。临时性文档（如 Bug 分析）完成后应归档。
2.  **更新**: 代码变更若影响功能，必须同步更新相关文档（尤其是 API 和 部署文档）。
3.  **审查**: 每月进行一次文档审查，删除无用文档，归档过时文档。
4.  **归档**: 确定不再具备当前参考价值但有历史价值的文档，移动到 `docs/archive/`。
5.  **删除**: 确定完全无用的临时文件（如 `plan_YYYYMMDD.md`）应直接删除。

## 4. 格式规范

*   使用标准的 Markdown 格式。
*   代码块必须指定语言（如 \`\`\`javascript）。
*   引用项目内文件使用相对路径。

## 5. AI 助手协作规则

*   AI 生成的临时计划文件 (`plan_*.md`) 默认存放在 `.trae/documents/`，任务完成后应自动清理或归档。
*   AI 在生成重要技术方案时，应直接在 `docs/architecture/` 下创建。
